Privacy & movement data

A movement game runs on location — which is exactly why MovenRun's default posture is summaries over trails, on-device over uploaded, and labeled over silent.

What the current app does with location

  • Foreground GPS only. Location is read while a session screen is active. No background tracking, geofencing, or location-based notifications.
  • On-device state. Sessions, zones, XP, streaks, clubs — the current build stores progress locally (AsyncStorage). There is no account system and no server sync.
  • Summaries, not trails, in derived features. The route-trust history, signal passport, and shareable proof cards intentionally persist and share summary statistics only — no raw GPS points, no coordinates, no route paths, no map screenshots.
  • No network submission. The current app does not upload routes to any backend. The verification pipeline exists as separate infrastructure (below) and is not yet connected.

What is not collected

  • No contacts, no ad identifiers, no third-party analytics or tracking SDKs.
  • No wallet addresses (the app has no wallet), no payment data (no payments).
  • No inference of anyone else's location — rivals and clubs are simulated.

Planned server behavior — clearly separate

When route verification connects (a later phase), the design is: the app submits a route to POST /gps/submit; the backend validates it, computes distance and zone coverage, stores the route's lifecycle (status, hash, distance, verification result), and deduplicates repeats. Requests to write endpoints are authenticated by wallet signature in the current backend design — relevant only to the future opt-in ownership layer. Retention windows and deletion controls for any server-side movement data are Planned work that must land before the pipeline serves real users; they are design requirements, not current features.

Future privacy work

  • Route storage minimization (persist verification artifacts, not raw trails, wherever possible).
  • Explicit retention and deletion controls, surfaced in-product.
  • A published privacy policy reviewed by counsel before any account system launches — the docs will not improvise one.

Boundary to remember: everything the current app knows stays on your phone. Server-side behavior described here is infrastructure and plans, labeled as such.
