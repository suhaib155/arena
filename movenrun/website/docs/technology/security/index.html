Security & data

Security work landed before the features that need it — so the pipeline was hardened while it still had zero users.

Infrastructure — the measures below protect the backend; the current app runs on-device and has no server attack surface of its own.

Wallet-signature authentication

Backend write endpoints (/gps/submit, /zones/mint, /battles/declare) require proof that the caller controls the wallet address they claim: a signed message checked by middleware, with the authenticated address bound to the request body so nobody can submit routes or declare challenges on someone else's behalf. This matters only for the future opt-in ownership layer — the consumer app has no wallet today.

Request hardening

  • Rate limiting on write endpoints, per address and per IP.
  • Security headers via Helmet.
  • CORS allowlist — only known origins may call the API from browsers.
  • Secrets — signing keys and tokens live in environment/CI secrets; nothing is committed. The oracle key is rotatable by design.

Role-based contract permissions

The contract suites use role-based access control (minter, oracle, season, challenge, DAO roles). The V1 testnet deployment has documented wiring weaknesses — see Contract status — and the V2 proposal tightens roles and signature binding throughout.

Threat model, summarized

Primary threats and current posture
ThreatPosture
Fake movement (GPS spoofing, replays)Validation checks + hashing + dedup today; stronger attestation gated before value attaches. Not claimed unbeatable.
Impersonation on write APIsWallet-signature auth with body binding
Abuse volume (farming, flooding)Rate limits; once-per-day progress rules in-app
Contract exploitationV1 is testnet-only with defects documented rather than hidden; V2 redesign under review; independent audit is a release gate
Data exposureMinimal collection (below); no accounts; no third-party trackers

Known limitations are listed honestly in Risks & limitations. There is currently no dedicated responsible-disclosure channel; issues can be reported via the public GitHub repository.

Data architecture

  • On-device (current app): sessions, zones, XP, streaks, clubs, previews — AsyncStorage on your phone. Derived screens recompute on demand rather than persisting new stores.
  • Backend (infrastructure): a routes table holding lifecycle status, route hash, distance, and verification result; queue jobs in Redis. Designed to persist verification artifacts rather than raw trails wherever possible.
  • On-chain (experimental testnet): V1 records on Base Sepolia only — nothing tied to real users.
  • Boundaries: the app sends nothing to the backend today; the backend writes nothing to mainnet ever (it has no mainnet); retention/deletion controls are Planned and gate the pipeline's first real users.
