Mobile application

A single Expo/React Native app carries the whole current product: the movement loop, the territory previews, and the progression systems — all on-device.

Implemented — Expo SDK 51, React Native 0.74, Expo Router v3, TypeScript.

Stack and structure

  • Routing: Expo Router v3 file-based routes — a tab shell (Today, Clubs, Profile) plus ~30 feature screens (/move/* session flow, /zone/[id], /territory/*, /route/* trust screens, city-layer screens, /deed-showroom, /network/status).
  • State: Zustand stores persisted to AsyncStorage. Progress (XP, level, streak, zones, clubs, history) lives on-device; derived screens (recap, objectives, districts, missions) compute on demand so they need no migrations.
  • Design system: the Daylight Cartography tokens live in src/theme.ts (palette, type scale, spacing, radii, shadows, motion). Motion uses core Animated only — no animation libraries, no map SDK yet.

GPS session architecture

The session flow (move/index → move/session → move/summary → move/captured) records foreground GPS positions into a route model (coordinates, distance, duration, pace). Summaries and all derived features persist summary statistics, not raw trails — the review history, passport, and proof card are deliberately privacy-safe.

Local preview systems

Territory, defense, clubs, districts, rivals, city war, sponsor/event zones, missions, mastery, collections, and the deed showroom are deterministic on-device models over local state. Each preview screen carries its label in the UI. This architecture is intentional: the seams where mock models plug in are the same seams where verified backend data plugs in later.

Build and distribution

  • Development: standard Expo dev flow (SDK 51; device testing via Expo Go + tunnel).
  • APK builds: an EAS GitHub Actions workflow (preview profile) produces installable Android builds. Authentication uses a repository secret; no credentials live in the codebase.
  • SDK policy: Expo SDK upgrades are isolated, device-tested PRs — never casual bumps.

Historical note: the quest shell

Historical The earliest app was a daily-quest shell used to prove the build/release pipeline and on-device state plumbing. Its session flow and XP store were reused and bent toward the movement loop — the quest screens themselves are legacy scaffolding, not the product.
