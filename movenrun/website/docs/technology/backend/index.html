Backend, routes & verification

The pipeline that will one day decide “this walk really happened” exists today — built, tested, and deliberately not yet serving the app.

Infrastructure — Express + BullMQ + Drizzle/Postgres + Redis. Not connected to the consumer app.

API responsibilities

  • POST /gps/submit — accept a route for verification (wallet-signature authenticated, rate-limited).
  • GET /gps/verify/:id — report a route's lifecycle status and, when verified, its proof.
  • /zones/*, /battles/*, /users/* — zone-mint signatures, challenge declarations, and user reads for the future ownership layer (testnet-oriented; unused by the current app).

Route lifecycle

  1. Submit. The route enters a BullMQ queue — validation never blocks the HTTP request.
  2. Validate. The GPS worker applies sanity checks: plausible speed between consecutive points, GPS-accuracy thresholds, and duration bounds.
  3. Compute. Haversine distance; zone coverage via the H3 mapping; a deterministic SHA-256 route hash over the route payload.
  4. Persist. Status, hash, distance, and result are stored (Drizzle/Postgres) so verification survives process restarts — and server-side deduplication rejects a repeated route before any signing.
  5. Sign. For verified routes, an oracle service signs a proof binding wallet, route hash, and distance.

The oracle concept

An “oracle” here is simply the backend acting as a witness: it attests, with a server-held signing key, that a route passed verification. On-chain contracts can then require that attestation before acting. In the current V1 design the attestation is a classic signed message; the V2 proposal (under review) upgrades every signature to typed, replay-resistant EIP-712 data. The oracle key is rotatable by design.

Verification checks and honest limits

  • Speed checks catch vehicles and teleports; accuracy checks catch synthetic or degraded traces; duration bounds catch nonsense sessions.
  • Route hashing + dedup prevent replaying the same walk twice — per wallet and per payload.
  • Limits: none of this makes GPS spoofing impossible. A determined attacker with a motivated toolchain can fabricate plausible traces. That is exactly why verification reliability is a release gate — the anti-spoofing bar rises (device attestation, behavioral signals, cross-checks) before anything of value ever depends on a route. MovenRun does not claim its verification is complete or unbeatable.

Integration status

The consumer app does not submit routes yet; every user-visible trust feature today is a local preview. Connecting the two is the core of Phase 3 — and privacy work (retention/deletion controls) lands before the pipeline serves real users.
